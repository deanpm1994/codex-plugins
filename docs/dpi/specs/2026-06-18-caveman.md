# Caveman DPI Handoff

Date: 2026-06-18

## Session Goal

Design a new local Codex plugin called `caveman` for this personal plugin repository. The goal is to reduce token usage by compressing agent communication while preserving enough meaning to keep work accurate.

## Context

This repository is a personal Codex plugin repo. The `caveman` work should live here as a standalone plugin, not as a wrapper CLI and not as an add-on to `example-plugin`.

The design conversation also confirmed that the current repository already contains unrelated in-progress changes under `plugins/example-plugin/`. Those changes should be left alone.

## Key Decisions

- Create a standalone `caveman` plugin.
- Keep the implementation plugin-only for now. Do not add a `codex --caveman ...` wrapper command.
- Support three levels: `low`, `medium`, and `high`.
- Support user activation and deactivation.
- Make the default level persistent, with per-session overrides allowed.
- Include a status surface for Codex statusline or app UI if available. If only one surface is feasible, prioritize the CLI/statusline surface first.
- Show current mode plus savings information in the status surface.
- Include companion functionality for stats and status, rather than forcing everything into a single skill file.
- Measure savings from actual usage when the platform exposes the needed data, and fall back to estimates otherwise.
- Store settings and stats persistently when possible, with a local fallback if persistent Codex storage is unavailable.

## Intended Behavior

- `caveman` should compress repetitive assistant output and selected internal notes.
- `low` should be conservative and mostly change phrasing.
- `medium` should compress more repeated text and boilerplate.
- `high` should be the most aggressive mode while still remaining usable.
- Users should be able to turn caveman on, off, and change levels from within Codex.

## Design Constraints

- Keep the plugin local to this repository.
- Prefer small, composable skills over a single large implementation file.
- Do not break normal tool use, task execution, or critical reasoning.
- Avoid claiming exact savings unless the plugin can actually measure them.

## Open Questions For Implementation

- What Codex APIs or plugin hooks are available for persistent storage?
- What Codex surface is available for statusline integration?
- What actual usage data can be read to compute savings?
- How should the plugin structure split between the main caveman skill and the stats/status companion skills?
